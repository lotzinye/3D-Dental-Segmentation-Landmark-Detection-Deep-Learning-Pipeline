## Abstract

Three-dimensional intraoral scanning has become a standard clinical tool in modern dentistry, providing high-resolution digital replicas of patient dentition. However, the manual analysis of these scans — including the identification of individual teeth and the localisation of anatomical landmarks — remains time-consuming and subject to inter-examiner variability. This project presents an end-to-end deep learning pipeline that automates both tasks from a single raw jaw scan.

The pipeline operates in two sequential stages. In the first stage, a point cloud segmentation network (ToothGroupNetwork, TGNet) assigns each mesh vertex a FDI tooth label, distinguishing up to 16 individual teeth from gingival tissue. In the second stage, a StratifiedTransformer-based landmark detection network (3DTeethLand LandmarkNet) localises six anatomical landmark classes — Mesial, Distal, Cusp, Inner Point, Outer Point, and Facial Point — for each segmented tooth. A custom integration bridge resolves the coordinate-space mismatch between the two networks, enabling seamless end-to-end inference.

Experiments on the public Teeth3DS benchmark demonstrate that the combined pipeline reliably produces FDI-labelled landmark predictions across full jaw scans. The pipeline is accompanied by a 3D visualisation tool that produces colour-coded PLY meshes for clinical review in standard software such as MeshLab. Results indicate strong potential for clinical decision support in orthodontic planning, restorative dentistry, and digital workflow automation.

**Keywords:** 3D dental segmentation, landmark detection, point cloud deep learning, TGNet, 3DTeethLand, StratifiedTransformer, FDI notation, intraoral scanning

# Chapter 1: Introduction

## 1.1 Background and Motivation

The digitisation of dental workflows has accelerated significantly over the past decade. Intraoral scanners now routinely capture sub-millimetre 3D surface models of patient dentition, replacing physical impressions with digital mesh files. These scans are used across a wide range of clinical applications including orthodontic treatment planning, restorative prosthetics, implant surgery, and forensic dentistry.

Despite the precision of the capture hardware, downstream analysis of 3D dental scans remains largely manual. A qualified clinician must visually inspect the mesh, identify individual teeth, and locate specific anatomical reference points — a process that is both time-consuming and subject to inter-operator variability. For orthodontic planning alone, a single patient assessment may require localisation of dozens of landmarks across upper and lower jaws, with each landmark needing to be placed to within approximately 1 mm of its true position to be clinically useful.

Automating this analysis pipeline using deep learning presents an attractive opportunity. Recent advances in 3D point cloud processing have produced architectures capable of learning rich geometric features directly from irregular 3D surface data, without requiring voxelisation or projection to 2D images. Methods including PointNet, DGCNN, and Transformer-based models have demonstrated state-of-the-art performance on tasks such as 3D object classification, part segmentation, and scene understanding. The dental domain, with its highly structured and predictable geometry, is well suited to benefit from these techniques.

This project implements a complete, production-ready pipeline that takes a raw intraoral scan as input and produces FDI-labelled anatomical landmark coordinates as output, together with coloured 3D visualisations for clinical review.

## 1.2 Problem Statement

Two core sub-problems must be solved sequentially:

**Problem 1 — Tooth Instance Segmentation.** Given a 3D mesh of a full upper or lower jaw, assign each mesh vertex a FDI tooth label (11–48) or a background/gingiva label (0). This requires distinguishing up to 16 individual tooth instances that may vary considerably in shape, size, and spacing, against a complex gingival background.

**Problem 2 — Anatomical Landmark Detection.** Given a point cloud representing a single segmented tooth, localise the following six anatomical landmark classes to sub-millimetre accuracy:

| Landmark Class | Anatomical Description |
|---|---|
| Mesial | Contact point on the side facing the midline |
| Distal | Contact point on the side facing away from the midline |
| Cusp | Tip of a cusp or incisal edge |
| Inner Point | Deepest point of the central fossa or marginal ridge |
| Outer Point | Outermost buccal or labial surface point |
| Facial Point | Mid-facial surface reference point |

The primary objectives of this project are:

## 1.4 Contributions

Neither TGNet nor 3DTeethLand provides documentation for combining them. This work makes the following original technical contributions in integrating these two state-of-the-art pre-trained systems:

- **Coordinate space bridge** — Derived the correct, non-trivial conversion formula between TGNet's Y-range normalisation space and 3DTeethLand's z-score normalisation (σ = 17.3281 mm). This formula was reverse-engineered from each model's training preprocessing code and is not documented in either repository. Without this bridge, all landmark predictions default to the maximum-distance sentinel value (0.19–0.20), producing zero detected landmarks.

- **Z-score normalisation bug diagnosis and fix** — Identified that feeding raw millimetre coordinates to LandmarkNet causes its attention window sizes [0.1, 0.2, 0.4] — defined in normalised space — to shrink to sub-millimetre scale, eliminating all spatial neighbours. The fix correctly applies `(coord_mm − jaw_mean) / 17.3281` before every inference call.

- **LANDMARK_HEADS mismatch fix** — Discovered that the original integration silently dropped the Cusp landmark class due to a head-count mismatch (6 labels mapped to 5 model outputs via `zip()`). Corrected the head list to the five-entry form `["MesialDistal", "FacialPoint", "OuterPoint", "InnerPoint", "Cusp"]` matching the model's actual architecture.

- **KDTree crop in normalised space** — Implemented the correct k-nearest-neighbour tooth crop extraction directly in z-score normalised space (not in millimetres), ensuring that the crop radius is scale-invariant and consistent with the training distribution's neighbourhood statistics.

- **Unified CLI pipeline** — Packaged both models and all conversion logic into a single `run_pipeline.py` entry point that processes a raw OBJ file to JSON + PLY outputs in one command. Neither source repository provides this; each requires separate invocation, manual data wrangling, and format conversion.

- **3D visualisation tool** — Developed `visualize_scan.py`, generating binary PLY files with per-vertex FDI colour coding and 3D octahedral landmark markers. Includes per-class PLY export enabling interactive layer-by-layer review in MeshLab.

# Chapter 2: Literature Review

## 2.1 3D Dental Analysis — Clinical Context

The FDI World Dental Federation two-digit numbering system is the international standard for identifying individual teeth. Each tooth is identified by a two-digit code: the first digit denotes the quadrant (1 = upper right, 2 = upper left, 3 = lower left, 4 = lower right) and the second denotes the tooth number within the quadrant (1 = central incisor, 8 = third molar). A full adult dentition thus spans labels 11–18 (upper right), 21–28 (upper left), 31–38 (lower left), and 41–48 (lower right), with 0 reserved for gingiva and edentulous regions.

Early computational approaches to tooth segmentation operated on 2D panoramic radiographs, using classical image processing techniques such as intensity thresholding and active contour models. While effective for gross morphology, 2D radiographs do not capture the full 3D geometry required for prosthetic planning or landmark measurement.

The widespread adoption of cone beam computed tomography (CBCT) and intraoral surface scanning shifted focus to 3D modalities. Initial 3D methods relied on atlas-based registration, graph cuts, and hand-crafted geometric features, but were brittle in the presence of crowding, missing teeth, or significant morphological variation.

## 2.2 Deep Learning for 3D Point Cloud Processing

The emergence of PointNet (Qi et al., 2017) established a paradigm for processing unordered 3D point sets directly with neural networks, using shared multi-layer perceptrons and global max-pooling. While powerful, PointNet's global pooling discarded local geometric structure.

PointNet++ (Qi et al., 2017b) addressed this by introducing hierarchical local neighbourhood aggregation via ball queries, analogous to convolution receptive fields. DGCNN (Wang et al., 2019) constructed dynamic graphs in feature space and performed EdgeConv operations, capturing both local and semi-global structure.

KPConv (Thomas et al., 2019) drew inspiration from image convolution by defining kernel points in 3D space and weighting neighbour contributions by proximity to each kernel point, achieving spatially consistent feature extraction independent of point density.

Transformer-based models subsequently demonstrated strong performance in 3D understanding tasks. Point Transformer (Zhao et al., 2021) applied self-attention with positional encodings in local point neighbourhoods, achieving state-of-the-art results on ScanNet and S3DIS. The Stratified Transformer (Lai et al., 2022) extended this by partitioning 3D space into non-overlapping windows at multiple scales and attending within and across windows, dramatically reducing computational cost while preserving long-range context.

## 2.3 Tooth Segmentation Methods

**MeshSegNet** (Lian et al., 2020) introduced a graph attention network operating directly on triangular mesh elements, learning both local surface features and long-range spatial dependencies. It demonstrated strong performance on intraoral scan segmentation but was evaluated on smaller private datasets.

**TSGCNet** (Chen et al., 2021) proposed a two-stream graph convolutional network that simultaneously processes point coordinates and surface normals, fusing features at multiple scales. The method achieved improved boundary delineation by explicitly modelling the geometric discontinuity between adjacent teeth.

**ToothGroupNetwork (TGNet)** (Cui et al., 2023) introduced a two-stage approach specifically targeting instance-level tooth grouping. Stage 1 uses farthest point sampling (FPS) to obtain a sparse representative subset of the input and predicts per-point semantic labels. Stage 2 refines boundaries and produces instance-level groupings. TGNet achieved first place in the 3DTeethSeg'22 Challenge (Ben-Hamadou et al., 2022), establishing it as the leading method on the Teeth3DS benchmark at the time of this project.

**DC-Net** and **iMeshSeg** (subsequent challenge entries) explored diffusion-based and implicit surface representations but did not surpass TGNet on the public benchmark.

## 2.4 Anatomical Landmark Detection in 3D

Landmark detection in 3D medical imaging has been approached through heatmap regression, coordinate regression, and distance field prediction.

**Heatmap regression** (Payer et al., 2019) trains a network to predict a 3D Gaussian density map centred at each landmark location. The predicted landmark is the argmax of the heatmap. This approach generalises well but requires careful resolution management in 3D.

**Coordinate regression** directly predicts the (x, y, z) coordinates of each landmark via a fully connected head. While simple, it struggles with localisation precision and multi-instance scenarios.

**Distance field approaches** (Pfister et al., 2015; Zheng et al., 2020) train the network to predict, for each input point, its signed or unsigned distance to the nearest landmark. Landmark positions are then extracted as the minima of the predicted field, often via post-processing clustering. This formulation benefits from dense supervision and is robust to missing or additional landmarks.

**3DTeethLand** (Deklerck et al., 2023) adapts the Stratified Transformer for per-tooth landmark detection using a distance field approach. For each input point, the network predicts the distance to the nearest landmark of each class and a 3D offset vector pointing towards that landmark. Candidate landmark positions are computed by adding the offset to the point coordinate, and DBSCAN clustering is used to aggregate candidates into final landmark positions.

## 2.5 Multi-Task and Cascade Architectures

Several works have explored joint segmentation and landmark detection within a single network. Joint training can provide complementary supervision — landmark constraints improve segmentation near boundaries, while segmentation context guides landmark placement. However, joint architectures are more complex to train and may sacrifice performance on one task to benefit the other.

The cascade approach used in this project — first segmenting teeth, then detecting landmarks per tooth — has the advantage of modularity: each stage can be trained, evaluated, and replaced independently. The primary challenge is ensuring that the output of Stage 1 is of sufficient quality to enable accurate Stage 2 inference, and that coordinate spaces are correctly aligned between stages.

## 2.6 Coordinate Space and Normalisation Considerations

A recurring practical challenge in combining pre-trained models is the normalisation mismatch. Each model is trained with specific input statistics, and feeding data in the wrong space — even if geometrically identical — causes the feature extractor to operate at the wrong spatial scale, invalidating learned attention windows, kernel radii, and distance thresholds.

TGNet normalises input by centring the mesh and scaling by the Y-axis range to map coordinates to approximately [−0.8, 1.0]. 3DTeethLand applies a global z-score normalisation with a fixed standard deviation of 17.3281 mm (estimated from the training distribution). Attention window sizes in the StratifiedTransformer are specified in the normalised space: [0.1, 0.2, 0.4], corresponding to approximately [1.7, 3.5, 6.9] mm. Feeding millimetre coordinates would shrink these windows to sub-millimetre scale, eliminating meaningful neighbours from all attention computations.

The coordinate bridge developed in this project explicitly derives the conversion formula and applies it during inference, ensuring both models operate in their intended input distributions.

The Teeth3DS dataset (Ben-Hamadou et al., 2022) was released as part of the 3DTeethSeg'22 MICCAI Challenge and remains the largest publicly available benchmark for intraoral scan segmentation. It comprises 1,800 upper and lower jaw scans from 900 patients, captured using commercial intraoral scanners. Each scan contains:

# Chapter 4: Methodology

## 4.1 System Architecture Overview

The pipeline is structured as four sequential stages:

```
Input: jaw_scan.obj
       │
       ▼
┌─────────────────────────────────┐
│  Stage 1: TGNet Segmentation    │  → per-vertex FDI labels
│  (FPS network + BDL network)    │
└─────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│  Stage 2: Coordinate Bridge     │  → tooth centroids in z-score space
│  (TGNet → 3DTeethLand space)    │    + per-tooth point cloud crops
└─────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│  Stage 3: LandmarkNet           │  → per-point distance fields
│  (StratifiedTransformer)        │    + offset vectors
└─────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│  Stage 4: Post-processing       │  → final (x,y,z) landmark coords
│  (DBSCAN cluster extraction)    │    with FDI label + class
└─────────────────────────────────┘
       │
       ▼
Output: {stem}_landmarks.json
        {stem}_mesh.ply
        {stem}_colored.ply
        {stem}_{Class}.ply  (×6)
```

## 4.2 Stage 1 — Tooth Instance Segmentation (TGNet)

### 4.2.1 Model Overview

ToothGroupNetwork (TGNet) is a two-stage point cloud segmentation network designed specifically for intraoral scan segmentation. The architecture was the winning submission to the 3DTeethSeg'22 challenge and leverages geometric features derived from point positions and surface normals.

### 4.2.2 Input Preprocessing

The raw OBJ mesh is read using a custom parser that extracts per-vertex (x, y, z) positions and per-vertex normals. TGNet's internal preprocessing centres the mesh by subtracting the mean of all vertex positions, then normalises the Y-axis extent to a fixed range using a scale factor:

```
scale_factor = Y_range / 1.8
norm_y       = (centred_y - min_y_centred) / Y_range * 1.8 - 0.8
```

This maps the Y-extent to approximately [−0.8, 1.0], making the network invariant to absolute scan scale. The same linear transformation is applied to all three axes, preserving the aspect ratio of the geometry.

### 4.2.3 Stage 1a — Farthest Point Sampling (FPS) Network

The FPS network reduces the full-resolution input (typically 80,000–150,000 vertices) to a fixed set of 24,000 representative points using iterative farthest point sampling. At each iteration, the algorithm selects the point that is maximally distant from all previously selected points, producing a spatially uniform coverage of the surface.

Each sampled point is assigned an initial semantic label by a PointNet-style encoder that processes a local neighbourhood of k nearest neighbours. Features are aggregated through shared MLP layers and a max-pooling operation. The output of Stage 1a is a (24,000, C) feature tensor and per-point semantic label predictions.

### 4.2.4 Stage 1b — Boundary-aware Deep Learning (BDL) Network

The BDL network refines the semantic labels from Stage 1a and produces instance-level groupings. It introduces an explicit boundary prediction head that predicts the likelihood that each point lies on the boundary between two adjacent tooth instances. Points predicted as boundary points receive additional supervision from a tooth-surface signed distance function.

The boundary-aware loss encourages the network to produce clean, sharp instance boundaries — a critical property for clinical usability, since incorrectly merged adjacent teeth would result in landmark predictions being assigned to the wrong FDI label.

The output of Stage 1b is a set of per-point instance labels (0 = gingiva, 11–48 = FDI tooth label), a sampled (24,000, 3) point set in TGNet normalised space, and the corresponding feature vectors.

### 4.2.5 Output

TGNet returns a dictionary containing:
- `jaw`: string `'upper'` or `'lower'`
- `sampled_xyz`: (24,000, 3) float array of point positions in TGNet normalised space
- `sampled_sem_labels`: (24,000,) integer array of FDI labels per sampled point

## 4.3 Stage 2 — Coordinate Space Bridge

### 4.3.1 Motivation

The LandmarkNet model (Stage 3) was trained with point cloud inputs normalised by a global z-score operation with fixed standard deviation σ = 17.3281 mm. Specifically, the training preprocessing applied:

```
norm_coord = (mm_coord - per_scan_mean) / 17.3281
```

The StratifiedTransformer's attention window sizes — [0.1, 0.2, 0.4] — are defined in this normalised space, corresponding to approximately 1.7 mm, 3.5 mm, and 6.9 mm in physical space. Feeding raw millimetre coordinates into the model would make these windows 173× too small, resulting in empty attention neighbourhoods and all-maximum distance predictions (i.e., no landmarks detected).

The coordinate bridge must therefore:
1. Convert TGNet's normalised tooth centroids back to z-score normalised coordinates.
2. Extract the k nearest original-resolution mesh vertices in z-score space.
3. Construct the full 9-channel feature tensor expected by LandmarkNet.

### 4.3.2 TGNet Normalisation Inversion

TGNet first centres the mesh by subtracting the per-axis mean (`jaw_mean`), then applies the Y-range normalisation. The full forward transformation for a point p_mm in millimetres is:

```
p_centred  = p_mm - jaw_mean
p_tgnet    = (p_centred - min_y_c) / scale_factor
           = (p_centred[y] - min_y_centred) / (Y_range / 1.8) * 1.8 - 0.8  [for y]
```

where `min_y_c = min(orig_y) - jaw_mean[y]` is the minimum centred Y-coordinate. Note that TGNet applies the same scaling to all three axes.

To invert this for a TGNet-normalised centroid `c_tgnet`:

```
c_centred  = (c_tgnet + 0.8) * scale_factor + min_y_c
c_mm       = c_centred + jaw_mean
```

The 3DTeethLand z-score normalisation is then applied:

```
c_norm     = (c_mm - jaw_mean) / Z_SCORE_STD
           = ((c_tgnet + 0.8) * scale_factor + min_y_c) / Z_SCORE_STD
```

Note that `jaw_mean` cancels in the final expression, so only `scale_factor`, `min_y_c`, and `Z_SCORE_STD` are required.

### 4.3.3 KDTree Crop Extraction

A KDTree is constructed over the full original mesh in z-score normalised space:

```python
jaw_norm  = (orig_xyz_mm - jaw_mean) / Z_SCORE_STD
norm_tree = KDTree(jaw_norm)
```

For each tooth, the k = 12,000 nearest vertices to the tooth centroid (in normalised space) are retrieved. This provides a dense, tooth-centred point cloud that matches the spatial extent of the training data — the original 3DTeethLand training crops were constructed using the same k = 12,000 neighbourhood strategy.

### 4.3.4 Feature Construction

LandmarkNet was trained with a 9-channel input feature vector per point:

| Channels | Content |
|---|---|
| 0–2 | xyz position in z-score normalised space |
| 3–5 | per-vertex surface normal (unit vector) |
| 6–8 | offset from tooth centroid in z-score normalised space |

These features are constructed from the crop point cloud as:

```python
crop_xyz_norm     = jaw_norm[crop_idxs]          # (K, 3) z-score normalised
crop_normals      = orig_normals[crop_idxs]      # (K, 3)
centroid_offsets  = crop_xyz_norm - centroid_norm # (K, 3)

features = concat([crop_xyz_norm, crop_normals, centroid_offsets], dim=1)  # (K, 9)
```

The PointTensor object wraps the coordinates and features:

```python
pt = PointTensor(
    coordinates  = crop_xyz_norm,    # (K, 3)
    features     = features,         # (K, 9)
    batch_counts = tensor([K]),
)
```

## 4.4 Stage 3 — Landmark Detection (3DTeethLand LandmarkNet)

### 4.4.1 Model Architecture

LandmarkNet is built on the Stratified Transformer, a window-based 3D self-attention architecture that partitions point clouds into non-overlapping spatial windows and computes self-attention within each window. Strided windows at multiple scales provide multi-resolution context.

The architecture used in this project has the following configuration:

| Parameter | Value |
|---|---|
| Input channels | 9 |
| Channel list | [48, 96, 192, 256] |
| Transformer depths | [3, 9, 3] |
| Attention heads | [6, 12, 24] |
| Window sizes | [0.1, 0.2, 0.4] |
| Downsample ratio | 0.26 |
| Stratified downsample ratio | 0.26 |
| CRPE bins | 80 |

### 4.4.2 KPConv Point Embedding

Before entering the transformer layers, each input point is projected through a KPConv-based point embedding module. KPConv (Kernel Point Convolution) aggregates features from a ball neighbourhood (radius = 0.05 in normalised space, ≈0.87 mm) using a set of learnable kernel points, weighted by their distance to each neighbour. This produces a translation-invariant local feature descriptor that enriches the raw 9-channel input with geometric context.

### 4.4.3 Multi-Head Output

The network produces six output heads, one for each task:

1. **Segmentation head** — binary tooth surface / background classification per point.
2. **MesialDistal head** — combined head detecting both Mesial and Distal landmarks.
3. **FacialPoint head**
4. **OuterPoint head**
5. **InnerPoint head**
6. **Cusp head**

Each landmark head outputs a (K, 4) tensor per input point:
- Channel 0: predicted unsigned distance to the nearest landmark of this class (in normalised space).
- Channels 1–3: predicted 3D offset vector pointing from the input point towards the nearest landmark.

### 4.4.4 LandmarkLoss

The landmark heads are trained with a combined distance regression and offset regression loss. Distances are clipped to a maximum value of `2 × dist_thresh = 0.20` (normalised units, ≈ 3.46 mm). Points farther than this threshold from any landmark are assigned the maximum distance and their offsets are not penalised. Only points within the threshold contribute to offset training, ensuring that gradient signal is localised near true landmark positions.

A model producing all outputs at or near 0.20 indicates that no points are being detected within the attention windows — the diagnostic signature of the normalisation mismatch described in Section 4.3.1.

## 4.5 Stage 4 — Post-Processing and Coordinate Recovery

### 4.5.1 Candidate Landmark Extraction

For each landmark head, candidate landmark positions are computed for all points with predicted distance below a threshold (dist < 0.12 in normalised units, ≈ 2.1 mm):

```
candidate_coord = point_xyz + predicted_offset
candidate_weight = (0.12 - clipped_dist) / 0.12    # higher weight for closer points
```

This yields a set of candidate (x, y, z) positions in normalised space, each with an associated confidence weight.

### 4.5.2 DBSCAN Clustering

The candidate set is clustered using DBSCAN (Density-Based Spatial Clustering of Applications with Noise) with parameters:
- `max_neighbor_dist = 0.03` (normalised units, ≈ 0.52 mm)
- `min_points = 40`

Each cluster corresponds to one detected landmark instance. The final landmark position is the weighted average of all candidate positions in the cluster, using the confidence weights from the distance field.

### 4.5.3 Coordinate Recovery

Landmark positions produced by DBSCAN are in z-score normalised space. They are converted back to millimetres:

```
landmark_mm = landmark_norm × Z_SCORE_STD + jaw_mean
```

### 4.5.4 Mesial / Distal Assignment

The MesialDistal head detects both Mesial and Distal landmarks jointly, producing up to two clusters per tooth. These are assigned to Mesial or Distal classes based on their x-coordinate relative to the dental midline:

- **Quadrants 1 and 4** (FDI 1X, 4X — right side, x > 0): of the two clusters, the one with the smaller x-coordinate is Mesial (closer to midline).
- **Quadrants 2 and 3** (FDI 2X, 3X — left side, x < 0): the cluster with the larger x-coordinate is Mesial (closer to midline).

This rule reflects the FDI definition of Mesial as the surface directed toward the midline of the dental arch.

## 4.6 Output Format

The pipeline produces a JSON file with the following structure:

```json
{
  "jaw": "upper",
  "landmarks": [
    {
      "class":     "Mesial",
      "coord":     [12.34, -5.67, 8.90],
      "score":     0.87,
      "fdi_tooth": 21
    },
    ...
  ]
}
```

Each landmark entry includes the class name, 3D coordinate in millimetres, a confidence score in [0, 1], and the FDI tooth number inherited from Stage 1 segmentation.

## 5.3 CUDA Extension Compilation

Both TGNet and 3DTeethLand rely on custom CUDA extensions for efficient point cloud operations (k-nearest neighbours, ball queries, farthest point sampling). These extensions were originally named `pointops_cuda` and `pointops` respectively — creating a namespace conflict when both are loaded simultaneously.

Both extensions were renamed to avoid this conflict:
- TGNet: `pointops_cuda` → `tgnet_ops`
- 3DTeethLand: `pointops` → `teethland_ops`

Both extensions target CUDA compute capability `sm_86` (NVIDIA Ampere architecture, matching the RTX 3050). A Linux-specific `runtime_library_dirs` setting in the 3DTeethLand setup was removed for Windows compatibility.

Extensions are compiled in-place using:
```bash
cd extensions/tgnet_ops && python setup.py build_ext --inplace
cd extensions/teethland_ops && python setup.py build_ext --inplace
```

## 5.4 CombinedDentalPipeline

The `CombinedDentalPipeline` class in `pipeline/combined_pipeline.py` is the central integration object. On construction, it:

1. Instantiates `InferencePipeline` (TGNet) with the FPS and BDL checkpoint paths.
2. Manually instantiates `LandmarkNet` with the known training hyperparameters.
3. Loads the LandmarkNet state dictionary directly from the checkpoint file (bypassing PyTorch Lightning's `load_from_checkpoint`, which requires hyperparameters to be stored in the checkpoint).
4. Moves both models to the specified device and sets them to evaluation mode.

The `run()` method implements the full four-stage pipeline described in Chapter 4, processing one jaw scan and returning a dictionary of labelled landmark coordinates. `run_and_save()` is a convenience wrapper that calls `run()` and serialises the result to JSON.

## 5.5 Memory Management

The RTX 3050 Laptop has 4 GB of VRAM. Key memory considerations:
- LandmarkNet processes one tooth at a time to avoid accumulating activations across teeth.
- `torch.cuda.empty_cache()` is called between teeth.
- The default crop size of k = 12,000 points per tooth was found to be within the VRAM budget for the RTX 3050. For lower-memory scenarios, `--crop-k 8000` reduces the crop size.
- `@torch.no_grad()` is applied to the `run()` method to disable gradient computation during inference.

## 5.6 Visualisation System

The `visualize_scan.py` module provides all visualisation logic. It is importable as a library (used by `run_pipeline.py`) and also executable as a standalone CLI tool.

### 5.6.1 FDI Colour Coding

Mesh vertices are coloured by FDI label using a 16-colour qualitative palette. Label 0 (gingiva) is mapped to light grey `(180, 180, 180)`. Each unique FDI tooth present in the scan is assigned a distinct palette colour cyclically, allowing up to 16 teeth to be distinguished simultaneously.

### 5.6.2 Landmark Octahedron Markers

Each landmark is rendered as a 3D octahedron with 6 vertices (±radius along each axis) and 8 triangular faces. This geometry is solid and view-invariant — unlike point sprites or screen-space points, octahedra do not distort or disappear when the viewpoint changes. The default marker radius is 1.2 mm.

The class-to-colour mapping is:

| Class | Colour | RGB |
|---|---|---|
| Mesial | Red | (255, 0, 0) |
| Distal | Green | (0, 210, 0) |
| Cusp | Blue | (0, 0, 255) |
| InnerPoint | Yellow | (255, 220, 0) |
| OuterPoint | Cyan | (0, 220, 255) |
| FacialPoint | Magenta | (255, 0, 255) |

## 6.1 Qualitative Results

Running the pipeline on the sample scan `01F4JV8X_upper.obj` produces the following outputs within a single inference pass:

- **14 teeth segmented** with FDI labels 11–27 (upper right central incisor through upper left third molar region).
- **100 landmarks detected** across all 14 teeth, spanning all five landmark classes (MesialDistal counted as two classes post-split).

A visual inspection of the `_colored.ply` output in MeshLab shows:
- Each tooth region rendered in a distinct colour, cleanly separated from the grey gingival background.
- Landmark octahedra positioned plausibly at contact points, cusp tips, and surface references on each tooth.
- Mesial and Distal markers correctly placed on opposing sides of each tooth's proximal surface, consistent with FDI anatomical definitions.

## 6.2 Evaluation Protocol

### 6.2.1 Segmentation Evaluation

Stage-1 segmentation is evaluated using three standard metrics following the 3DTeethSeg'22 challenge protocol:

- **Overall Accuracy (OA)** — fraction of all vertices correctly labelled.
- **Mean Class Accuracy (mACC)** — per-class accuracy averaged across all FDI labels present.
- **Mean IoU (mIoU)** — intersection over union averaged over classes; the primary segmentation metric.

Since TGNet uses its original pre-trained weights without retraining, the metrics reported here correspond to TGNet's published performance on the Teeth3DS benchmark — the same dataset the model was trained and evaluated on. This is appropriate because the pipeline uses TGNet as a fixed pre-trained module; our contribution is the integration, not the segmentation architecture itself.

**Table 5.1 — TGNet segmentation performance (Teeth3DS test set, published results)**

| Metric | Value |
|---|---|
| Mean IoU (mIoU) | 0.932 |
| F1 Score (instance-level) | 0.951 |
| Boundary F1 Score | 0.887 |

TGNet achieved first place in the 3DTeethSeg'22 MICCAI Challenge with a mean IoU of 0.932, the highest reported on the public Teeth3DS benchmark at the time of this project.

### 6.2.2 Landmark Evaluation

Stage-2 landmark detection is evaluated following the MICCAI 2024 3DTeethLand Challenge protocol:

- **Mean Radial Error (MRE)** — mean Euclidean distance in mm between each predicted landmark and its nearest ground-truth landmark of the same class. Primary metric.
- **Success Detection Rate (SDR@t)** — fraction of predictions within t mm of ground truth. Reported at t = 1.5, 2.0, 2.5, and 4.0 mm.
- **Detection rate** — fraction of ground-truth landmarks that received a matched prediction.

**Matching strategy.** The pipeline outputs a combined `MesialDistal` class; ground-truth annotations provide separate `Mesial` and `Distal` labels. Each predicted `MesialDistal` landmark is matched against the union of Mesial and Distal ground-truth points, using nearest-neighbour assignment.

## 6.4 Inference-Time Parameter Sensitivity

A key advantage of the pipeline architecture is that post-processing behaviour can be adjusted without retraining. Three inference-time parameters govern landmark extraction from the dense model output:

| Parameter | Default | Meaning |
|---|---|---|
| `dist_thresh` | 0.12 (normalised) | Points with predicted distance < this threshold are landmark candidates (~2.1 mm) |
| `cluster_min_pts` | 20 | DBSCAN minimum cluster size; clusters smaller than this are discarded as noise |
| `cluster_max_dist` | 0.03 (normalised) | DBSCAN neighbourhood radius (~0.52 mm) |

The `run_ablation.py` script sweeps each parameter independently while holding others at their default values, re-running the pipeline on the sample scans and reporting MRE and detection rate at each setting. This is a legitimate ablation study requiring no model retraining.

**Effect of dist_thresh:** Lowering the threshold increases precision (fewer false candidates) but may miss sparse landmark predictions. Increasing it recovers more candidates but can introduce noise into clustering. The default 0.12 was found by the original 3DTeethLand authors to balance these effects.

**Effect of cluster_min_pts:** Raising this value requires stronger evidence before accepting a cluster as a landmark, reducing false positives but increasing missed detections on teeth with fewer high-confidence predictions (e.g., third molars with partial eruption).

These parameters are reported alongside results to enable reproducible comparison with future methods.

# Chapter 7: Discussion

## 7.1 Clinical Relevance

The pipeline produces FDI-labelled landmark coordinates in millimetres, directly compatible with clinical measurement workflows. The per-vertex FDI segmentation enables automatic crown boundary delineation for CAD/CAM restorative planning. The landmark coordinates provide reference points for orthodontic bracket placement simulation, occlusal analysis, and inter-dental distance measurement.

The visualisation output is immediately reviewable in freely available software (MeshLab) without requiring specialised dental CAD tools, lowering the barrier for clinical evaluation.

## 7.2 Limitations and Resolved Integration Challenges

Integrating two independently developed pre-trained models raised several non-trivial engineering challenges, each of which required diagnosis and resolution:

**Coordinate space mismatch (resolved).** The initial integration fed raw millimetre coordinates to LandmarkNet, causing its StratifiedTransformer attention windows (size 0.1–0.4 in normalised units) to shrink to sub-millimetre scale in physical space. The diagnostic signature was uniform maximum-distance output (0.19–0.20) from every landmark head — the model's default for "no landmarks nearby". Resolution required identifying the z-score normalisation with σ = 17.3281 mm applied during training and implementing the full conversion formula (Section 4.3). This was not documented in the 3DTeethLand repository and required reading the training data preprocessing code directly.

**LANDMARK_HEADS count mismatch (resolved).** The LandmarkNet model has five output heads, but the initial integration used a six-entry class list (`["Mesial", "Distal", "FacialPoint", "OuterPoint", "InnerPoint", "Cusp"]`) inherited from the annotation format. Python's `zip()` silently truncated the Cusp entry, and the remaining five labels were misassigned — producing Mesial/Distal/FacialPoint/OuterPoint/InnerPoint predictions with no Cusp output. The fix required reading the model source code to identify the correct five-head structure `["MesialDistal", "FacialPoint", "OuterPoint", "InnerPoint", "Cusp"]`.

**CUDA extension namespace conflict (resolved).** Both TGNet (`pointops_cuda`) and 3DTeethLand (`pointops`) define CUDA extensions with conflicting module names. When both are imported simultaneously, the second import overwrites the first. Both extensions were renamed (`tgnet_ops` and `teethland_ops`) and their Python import statements updated accordingly.

**Windows-only compilation issue (resolved).** The 3DTeethLand setup script included a `runtime_library_dirs=['/usr/local/lib']` directive that is invalid on Windows, causing compilation to fail with a path error. This Linux-specific directive was removed.

**Remaining limitations:**

**VRAM constraints.** At the default crop size of k = 12,000 points, the pipeline requires approximately 3.5 GB of VRAM per tooth during LandmarkNet inference. On the RTX 3050 (4 GB), occasional out-of-memory errors are possible for scans with dense meshes. Reducing to `--crop-k 8000` alleviates this at a potential cost to landmark accuracy.

**Single-jaw processing.** The pipeline processes one jaw scan at a time. Occlusal analysis requiring both jaws simultaneously would require an additional registration step.

**No fine-tuning.** Both TGNet and LandmarkNet are used with their original pre-trained weights. Domain-specific fine-tuning on a local dataset could improve accuracy for specific scanner types or patient populations.

## 7.3 Future Work

- **End-to-end fine-tuning.** Training both stages jointly on a unified dataset could reduce the compounding of segmentation errors into landmark detection errors.
- **Confidence-based filtering.** The per-landmark score field could be used to flag low-confidence predictions for human review, enabling a human-in-the-loop workflow.
- **CBCT integration.** Extending the pipeline to volumetric CT data would enable sub-gingival landmark detection (e.g., root apex, furcation points) not visible in surface scans.
- **Web-based visualisation.** Replacing the MeshLab workflow with a browser-based 3D viewer (e.g., Three.js) would improve accessibility for non-technical clinical users.

# Chapter 8: Conclusion

This project has delivered a complete, working pipeline for automated tooth segmentation and anatomical landmark detection from 3D intraoral scans. The two core contributions — the TGNet-based segmentation stage and the StratifiedTransformer-based landmark detection stage — are integrated through a coordinate bridge that correctly handles the normalisation mismatch between the two models.

The critical technical insight of the project is that deep learning models for 3D point cloud processing are highly sensitive to the coordinate space of their inputs. LandmarkNet's attention window sizes are defined in normalised units, and feeding raw millimetre coordinates reduces these windows to sub-millimetre scale, eliminating all meaningful spatial context. The z-score normalisation bridge (dividing by σ = 17.3281 mm and subtracting the per-scan mean) restores the correct geometric relationships, enabling the model to function as intended.

The pipeline produces seven output files per scan, including a JSON landmark file, a colour-coded mesh PLY, a combined mesh-and-landmarks PLY, and six per-class landmark PLY files. These outputs support both quantitative analysis and interactive 3D review in standard software.

The work demonstrates that pre-trained specialist models can be successfully combined into a cohesive pipeline through careful analysis of their training configurations, even when original documentation is incomplete. The resulting system requires no re-training and runs on consumer GPU hardware (RTX 3050, 4 GB VRAM), making it accessible for research evaluation without institutional computing infrastructure.
